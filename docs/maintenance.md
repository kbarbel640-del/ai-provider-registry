# Maintenance Runbook

Step-by-step for keeping the registry in sync. The registry is **hand-maintained
and redundant**: the same information lives in parallel across the YAML
directories, `registry.json`, `llms.txt`, and `index.html` (which carries
hardcoded stat counts and link lists). Forgetting one of these desyncs the
registry. After every change below, run `python3 validate_registry.py -v`
(exit 1 on any mismatch) — and remember CI (`.github/workflows/validate.yml`)
runs the same check on every push/PR, so a desynced commit fails regardless of
who pushed.

## Add a provider

1. Create `providers/<id>.yaml` following `docs/provider-schema.md`. Use
   `kimi.yaml` as the template for rich per-model metadata
   (`context_window`, per-token `pricing`, `tags`). Set `pricing.free` truthfully
   — it is cross-checked against `registry.json` and the `index.html` badge.
2. Add the provider to the `providers` array in `registry.json`: `id`, `free`,
   `api_style`, `supports`, `api_spec`, and optional `endpoint` / `model_count`.
   Keep the array ordering in sync with `llms.txt` and `index.html` for
   readability (not enforced, but expected).
3. Append `providers/<id>.yaml` to the **Providers** section of `llms.txt`.
4. In `index.html`, add a `<li>` link in the **Providers** list with the matching
   `free tier: yes` / `free tier: no` badge, and bump the **Providers** stat
   count by one.
5. If the provider introduces a capability not yet in the `capabilities` array of
   `registry.json`, add it there. Give it a per-file YAML under `capabilities/`
   **only if** it is a real, documented capability (see "Add a capability"
   below). Otherwise add it to `REGISTRY_ONLY_CAPS` in `validate_registry.py`.
6. `api_spec` is either an absolute `https://` URL to a remote OpenAPI/Discovery
   spec **or** a repo-relative path (e.g. `api-specs/nvidia.md`); for the
   latter, the referenced file must exist (the validator checks this). If you
   vendor a spec, drop it under `api-specs/`.

## Add a model family

1. Create `models/<family>.yaml` following `docs/model-schema.md`. Note that a
   `variant` can be served by a **different** provider than the family's
   creator (e.g. `claude-opus-4.8` under provider `opencode`).
2. Append `models/<family>.yaml` to the **Models** section of `llms.txt`.
3. Add a `<li>` link in the **Models** list of `index.html` and bump the
   **Model Families** stat count by one.
4. (Models are **not** tracked in `registry.json`.)

## Add a capability

1. Create `capabilities/<id>.yaml` following `docs/capability-schema.md`
   (`description` + JSON `schema`).
2. Add the capability id to the `capabilities` array in `registry.json`.
3. Append `capabilities/<id>.yaml` to the **Capabilities** section of
   `llms.txt`.
4. Add a `<li>` link in the **Capabilities** list of `index.html` and bump the
   **Capabilities** stat count by one.

> Exception: a handful of capabilities (`video`, `audio`, `realtime`) exist
> **only** in the `registry.json` array with no per-file YAML, no `llms.txt`
> entry, and no `index.html` link — these are whitelisted via
> `REGISTRY_ONLY_CAPS` in `validate_registry.py`. Add a real capability the full
> way; only use the registry-only form if you deliberately want a capability id
> referenced by providers without its own YAML page, and then you **must** add it
> to `REGISTRY_ONLY_CAPS` or the validator will flag it.

## Remove an entry

Reverse each of the steps above for the relevant axis, including decrementing
the matching `index.html` stat counter and removing the `llms.txt` /
`index.html` / `registry.json` references. Then run the validator.

## OSINT registry

`osint-providers/` is a separate registry generated by
`split_osint_providers.py` from source YAML **not** checked into this repo (see
`CLAUDE.md`). For corrections, edit the per-file YAMLs directly. The validator
does not cross-check the OSINT tree.
